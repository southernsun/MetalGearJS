# Metal Gear MSX2 Music + SFX Player

A C# WPF application that plays every music track and sound effect from Metal Gear MSX2 by emulating the AY-3-8910 PSG and re-implementing the BGM driver. The project folder is still called `ThemeOfTaraPlayer/` for historical reasons (Theme of Tara was the first track ported).

For deep dives on how the original sound system works and how the port mirrors it, see [`docs/`](../../docs/README.md).

> **Note:** the `sound/*.asm` paths cited below live in the separate MSX disassembly repo ([southernsun/MetalGear](https://github.com/southernsun/MetalGear), a sibling clone at `../MetalGear`). This player itself reads **no** `.asm` at runtime — its song/SFX data is baked into `MusicCatalog.cs` (regenerated by `extract_music.py` from `MetalGear.lst`), so it runs without the disassembly present and writes WAVs into this repo's `web/assets/`.

## Changelog

### 2026-05-24 — Extractor: short-form db parser
- Fixed `extract_music.py` regex that dropped any `db` line where sjasm packed the source text right after a 4-byte run with only a single-space separator (e.g. `00001+++7197 FE 00 22 01 Sfx_Laser: db …`). The previous heuristic split on 2+ whitespace and rejected the resulting `"FE 00 22 01 Sfx_Laser:"` for failing the pure-hex check, so the bytes were silently zeroed in the catalog. Now matches either concatenated 4..32-hex-char form or 1..4 single-space-separated `XX` chunks explicitly. This was the root cause of Sfx_Laser playing wrong — its 8 bytes were extracted as zeros.

### 2026-05-24 — SFX step-format fix (noise period + envelope period)
- **Fixed SFXs that put a noise-period byte between a config and a freq/vol pair** (Sfx_Laser, Sfx_Boomerang, Sfx_ShellWhistle, Sfx_SmgShot, Sfx_Click, Sfx_BarrelHit, Sfx_ShellShot, Sfx_MissileShot, Sfx_FireShotSup, Sfx_PunchGuard, etc.). The earlier port treated `0x2x`, `0x1x` and freq/vol as three mutually exclusive top-level branches; the ASM driver walks them as a sequence — `optional config + duration → optional env-period (if envelope mode) → optional noise period → freq/vol`. Refactored both `ProcessInstrumentData` and `ProcessSfxCommands` to share a single `ProcessSoundStep` helper that follows the ASM ChkCmd_2x → ChkCmd_1x → SfxLogic3 flow.
- **Added envelope-period reads** when a config byte has bit 3 set (envelope) and bit 2 clear (set-env-freq): the next two bytes go to PSG registers 12 (high) and 11 (low). This was previously consumed as a freq/vol pair, which silently corrupted instruments like SnareDrum.

### 2026-05-24 — SFX catalog + UI categories
- Extractor extended to cover the whole sound region (`0x6950..0x87B7`), pulling early SFXs, music, and late SFXs into one buffer.
- Added 38 single-channel SFXs (`Sfx_Propeller`..`Sfx_RankDown`) and 6 multi-channel SFXs (menu logo, pause, radio noise, Big Boss dies, ending explosion). Single-channel SFXs use `Sfx_Dummy` (`db 0FFh`) as a silent filler for channels B/C so they shut down on the first frame.
- `Song` record gained a `Category` field (`"Music"` or `"SFX"`); the ComboBox now shows `"Music — Theme of Tara"`, `"SFX — Bullet shot"`, etc.

### 2026-05-24 — Multi-song catalog
- **All songs available**: replaced the hand-written `ThemeOfTaraData.cs` with an auto-generated `MusicCatalog.cs` covering all 10 tracks (both Theme of Tara intro + main, Red Alert, Alert, Sneaking Mission, TX-55, Beyond Big Boss, Mercenary, Return of Fox Hunter, Just Another Dead Soldier).
- **`extract_music.py`**: parses `MetalGear.lst` (sjasm's listing file) and pulls out every assembled music byte plus all song-start addresses. The player treats the music as a buffer indexed by MSX RAM address, so the `0xFE … aabb` branch addresses inside the bytestream don't need rebasing.
- **Song selector UI**: `ComboBox` in `MainWindow.xaml` binds to `MusicCatalog.Songs`. Switching songs while playing hot-swaps.

### 2026-05-23 — ASM-parity pass
- **Fixed first-two-notes silence (PSG phase runaway)**: `PsgEmulator.GenerateSamples` only subtracted 1 from the phase accumulator per sample via `if`. With all period registers at 0 on reset (treated as `period=1`), each sample pushed phase up by ~2.5 and only dropped it by 1, so over the 28-frame opening rest of Theme of Tara the accumulator climbed to ~30,000+. When the first real period was finally written the phase stayed `>= 1.0` every sample, toggling at Nyquist (inaudible) for ~1 second before draining. Replacing `if` with `while` in the tone/noise/envelope generators keeps phase bounded in `[0, 1)` and the first notes play immediately.
- **Fixed ADSR decay**: rewrote `ProcessAdsrDecay` to match `bgmdriver.asm:407-435` — decrement `DecayCount` twice (and volume once) per frame during the decay window, then sustain until `DecayCount <= Release`, then release one volume step per frame. The previous `DecayCount % DecaySteps == 0` gate produced the wrong envelope shape entirely.
- **Fixed vibrato direction**: ASM `cpl` on `VIBRATO_CLK` makes the first cycle decrement frequency; C# was incrementing first.
- **Fixed silence frequency reset**: `0xCx` (rest) now zeroes `FREQ_LOW`/`FREQ_HIGH` and writes the PSG, matching ASM `MuteNote`.

### 2025-11-27
- **Channel synchronization fix**: `NoteCounter` was not being decremented while a channel was in instrument mode, causing channels to drift apart. It is now always decremented each frame, and when the note duration expires the channel immediately returns to note mode and processes the next command.
- **Instrument envelope fix**: Instrument data is a sequence of frequency/volume steps, each held for a duration set by the most recent config command. The duration is now stored in `InstrumentLength` and reapplied for every subsequent freq/vol step, so the full envelope plays out instead of cutting off after the first step.

### 2025-11-25
- **Instrument freq/vol byte order fix**: Instrument data byte order matches the original ASM:
  - Byte 1: `[Volume:4][FreqHigh:4]`
  - Byte 2: `[FreqLow:8]`

  The earlier port had the two halves swapped.
- **Debug panel** (`DebugInfo.cs`, MainWindow): per-channel mode/note/octave/freq/volume/instrument/tempo/config/last-command/vibrato/detune, global frame count and mixer register, per-channel note history, and a selectable command-execution log.

## Architecture Overview

### Components

```
ThemeOfTaraPlayer/
├── PsgEmulator.cs      - AY-3-8910 PSG chip emulation
├── SoundChannel.cs     - Channel state (mirrors Z80 SOUND structure)
├── MusicEngine.cs      - Music command processor (BGM driver port)
├── InstrumentData.cs   - Drum kit and bass instrument definitions
├── MusicCatalog.cs     - Auto-generated song table + music bytes (from extract_music.py)
├── extract_music.py    - Extractor that regenerates MusicCatalog.cs from MetalGear.lst
├── DebugInfo.cs        - Debug snapshot classes
├── AudioPlayer.cs      - NAudio-based audio output
├── MainWindow.xaml     - WPF UI with debug panel
└── MainWindow.xaml.cs  - UI code-behind
```

> The hand-written `ThemeOfTaraData.cs` was removed when the catalog became
> auto-generated — see the 2026-05-24 changelog entry and
> [`docs/tools/music-extraction.md`](../../docs/tools/music-extraction.md).

### How It Works

1. **PsgEmulator** generates audio samples at 44.1kHz by emulating the PSG chip
2. **MusicEngine** processes music commands at 60Hz (MSX2 frame rate)
3. **AudioPlayer** uses a custom `IWaveProvider` for smooth, on-demand sample generation
4. Each frame, the engine processes commands and updates PSG registers

## Original Sound System Analysis

### Hardware

- **Sound Chip**: AY-3-8910 PSG (Programmable Sound Generator)
- **Clock**: 1.7897725 MHz, divided by 16 for tone generation
- **Channels**: 3 tone channels + 1 noise generator
- **Frame Rate**: 60 Hz (NTSC)

### PSG Registers

| Register | Purpose |
|----------|---------|
| 0-1 | Channel 1 frequency (12-bit) |
| 2-3 | Channel 2 frequency (12-bit) |
| 4-5 | Channel 3 frequency (12-bit) |
| 6 | Noise period (5-bit) |
| 7 | Mixer (tone/noise enable per channel) |
| 8-10 | Volume for channels 1-3 (4-bit or envelope) |
| 11-12 | Envelope period (16-bit) |
| 13 | Envelope shape |

### Mixer Register (7)

- Bits 0-2: Tone enable (0 = enabled, 1 = disabled)
- Bits 3-5: Noise enable (0 = enabled, 1 = disabled)

### Music Command Format

The music uses a byte-stream command format:

| Command | Format | Description |
|---------|--------|-------------|
| `0x-Bx` | 1 byte | Play note (high nibble = note 0-11, low nibble = duration) |
| `Cx` | 1 byte | Silence/rest (low nibble = duration) |
| `Dx` | 1 byte | Set tempo (low nibble = tempo value) |
| `E0-E7` | 1 byte | Set octave (0-7) |
| `E8` | 1 byte | Enable detune (+1 to frequency) |
| `E9 xx` | 2 bytes | Select instrument |
| `EC xy` | 2 bytes | Vibrato (x=speed, y=intensity) |
| `EF` | 1 byte | Note OFF (clear modifiers) |
| `Fx yz` | 3 bytes | ADSR envelope (x=volume, y=decay, z=release) |
| `FE xx addr` | 4 bytes | Loop xx times to address |
| `FE FE addr` | 4 bytes | GOTO (infinite loop) |
| `FE FF addr` | 4 bytes | CALL subroutine |
| `FE 00` | 2 bytes | Toggle note/SFX mode |
| `FF` | 1 byte | END or RETURN from subroutine |

### Note Frequency Table (Octave 0)

Note indices start from C (standard):
```
C=0x6B, C#=0x65, D=0x5F, D#=0x5A, E=0x55, F=0x50
F#=0x4C, G=0x47, G#=0x43, A=0x40, A#=0x3C, B=0x39
```

Higher octaves: frequency is left-shifted by octave number.

### ADSR Envelope

- **Attack**: Immediate (set by Fx command, volume = low nibble + 1)
- **Decay**: Decrements over DecaySteps frames
- **Sustain**: Holds volume for remainder of note
- **Release**: Decrements over Release frames after note ends

### Instruments

**Instrument 1: Drumkit** (12 sounds by note value)
- Note 0 (C): Hi-hat
- Note 1 (C#): Cymbal
- Note 2 (D): Snare drum
- Note 3 (D#): Snare drum 2
- Note 7 (G): Tom 1
- Note 8 (G#): Tom 2
- Note 9 (A): Tom 3
- Note 10 (A#): Tom 4
- Note 11 (B): Bass drum

**Instrument 4: Slap Bass** (notes 4, 7, 9, 11)

**Instrument 5: Bass** (notes 0-2, 4-7, 9, 11)

Instruments use SFX mode with direct frequency/volume control.

### Instrument Data Format

Each instrument consists of SFX-mode commands:
- `0x2y duration` - Config byte (y: bit1=tone, bit0=noise) + duration in frames
- `0xVF 0xLL` - Volume (high nibble) + Freq high (low nibble), then Freq low byte
- `0xFF` - End of instrument

Example for Bass4 (instrument 5, note E):
```
0x22, 0x01,           Config: tone ON, duration 1 frame
0xC5, 0x4D,           Vol=12, Freq=0x54D
0xC2, 0xA6,           Vol=12, Freq=0x2A6
0x22, 0x04,           Config: tone ON, duration 4 frames
...continues with decaying volume...
0xFF                  End
```

### Channel Work Area Structure

Each channel has 32 bytes of state (from `constants/structures/sound.asm`):

```
Offset  Name              Description
+0      NOTE_COUNTER      Frames remaining for current note
+1      NOTE_LENGTH       Duration of current note
+2      ID                Sound/Music ID (0=silent)
+3-4    POINTER           Current position in music data
+5      CONFIG            Bit flags (envelope, tone, noise)
+6      TEMPO             Note length multiplier
+7      ATTACK_VOL        Initial volume for ADSR
+8      DECAY_VOL         Current decay volume
+9      OCTAVE            Current octave (0-7)
+10     DECAY_CNT         Remaining decay+release steps
+11     LOOP_CNT          Loop iteration counter
+12     DECAY_STEPS       Number of decay steps
+13     RELEASE           Release steps count
+14     NOTE_MODE         1=Note mode, 0=SFX mode
+15     FREQ_MOD          Detune/vibrato flags
+16     INSTRUMENT        Instrument ID
+17-18  INSTRUMENT_PTR    Pointer to instrument data
+19     INSTRUMENT_CNT    Instrument tick counter
+20     INSTRUMENT_LEN    Instrument note duration
+21-22  FREQ              Current frequency
+23     VOLUME            Current volume
+24-25  RETURN_ADDR       Return address for CALL
+26-28  VIBRATO           Wait, clock, config
```

## Theme of Tara Structure

The song consists of 3 channels:
- **Channel A (Melody)**: Main melodic line with bass instrument
- **Channel B (Harmony)**: Counter-melody with slap bass and drums
- **Channel C (Bass)**: Low-frequency bass line with drums

Uses 8 shared subroutines for memory-efficient pattern reuse.

### Headless audio export

Besides interactive playback, this tool renders the ROM's sounds to WAV for the
[`web/`](../../web/) browser port (dispatched in `App.xaml.cs`):

- `--export-punch`, `--export-door`, `--export-alert`, `--export-dead`, `--export-pickup`,
  `--export-spawn` — named one-off SFX exports.
- `--export-sfx "<catalog name>" [out.wav] [seconds]` — render any music track or SFX by its
  `MusicCatalog` name (the optional `seconds` fixes the render length for looping music).
- `--export-sfx-rel "<name>" "<refName>" [out.wav]` — render an SFX scaled *relative* to a
  reference sound's level instead of normalising it (preserves the ROM's quiet PSG sounds;
  see the note in [`docs/SESSION-STATE.md`](../../docs/SESSION-STATE.md)).

## Next Steps

1. **Compare against a real MSX emulator**: Capture PSG register writes from openMSX while playing Theme of Tara and diff against this player's writes frame by frame.
2. **Noise period scaling**: `PsgEmulator.GenerateSamples` doubles the noise period (`GetNoisePeriod() * 2`) on top of the doubling the original ASM does before writing register 6 (see `bgmdriver.asm:289-296`). On a real AY-3-8910 the noise rate is `master_clock / (16 * period)` with no extra `/2` stage, so the current implementation likely runs drum/percussion noise an octave too low. Worth A/B testing against openMSX.
3. **Pause/resume support**: `MusicFadeOut`/`VolumeFadeVal`/`SoundDataSaved` from the ASM driver are not yet ported. Only relevant if the player grows beyond Theme of Tara playback.
4. **Envelope-volume mode**: registers 11/12/13 are emulated, but the music engine never writes them for Theme of Tara — verify behaviour with a track that uses envelope volume (e.g. Sneaking Mission) before claiming completeness.
