# Browser port — session state / handoff

A browser re-creation of Metal Gear (MSX2) is being built slice-by-slice in `web/`, with all
assets decoded **from the disassembly** by headless export modes added to the C# tools. Plain
HTML/JS/Canvas, no build step. Planned and tracked via OpenSpec under `openspec/changes/`
(36 changes implemented, archived, and — through the 2026-06 batch — USER-PLAYTESTED);
ROM-routine coverage is tracked in
`docs/rom-coverage.md`, generated by `node Tools/coverage/coverage.mjs`.

> **The disassembly is a sibling repo.** The `.asm` sources every exporter/audit/coverage
> script reads live in [southernsun/MetalGear](https://github.com/southernsun/MetalGear),
> expected as a sibling clone at `../MetalGear` (override with `MG_ROM_DIR`). The export
> commands below need it; the playable `web/` game does not (it runs on the checked-in
> `web/assets/`).

The ROM's data
storage formats (graphics planes, door/item/radio/text tables, sound opcodes) are
documented in `docs/rom-data-formats.md`. Coverage is now 91% strict / 96% blended with
ZERO remaining `todo` routines — every in-scope routine is `done` or a documented `partial`
(behavioural stand-ins / cosmetic sprite tables / unmodelled-prerequisite gaps). Two small
final ports closed the sweep: ChkUsingArmor (the bullet-proof vest item 1 halves damage in
damage()) and ShowEmergencyText (Metal Gear's destruction shows text 150, unskippable).

## How to run / export
- Serve the game: `cd web && node serve.js` → open `http://localhost:8099`. The game boots
  through the title sequence (Konami logo line-reveal on white → the METAL GEAR logo swoop
  with its smear trail + SFX → "© KONAMI 1987 / PUSH SPACE KEY"); any key skips the boot to
  the finished title (ChkAnykeyStart), and Space or M starts the game — which plays the
  INTRO scene first (IntroSceneLogic: the scripted swim into room 121, Big Boss's call with
  text 2 over the radio UI, the fence climb + hop; the landing spot becomes the death
  checkpoint per ChkSaveGameStatus). The first keypress
  also unlocks audio (browser policy) — an untouched boot runs silent, like the ROM's muted
  logo phase. Dev hooks (?room= / #auto) skip the title.
- Controls: arrows/WASD move · M punch · Space fire · Q weapon menu · E item menu (in a menu,
  MOVING the cursor selects — no confirm key, like the ROM; Space in the item menu USES the
  selected item, e.g. ration = full heal) · R radio (the ROM's F4: ←/→ tune the frequency,
  ↑ = SEND; R also exits, even mid-text) · 1-7 select weapon, 0 holster · I cycle item ·
  P pause (the pause overlay shows the current ROOM number) · M or Enter advances + dismisses
  text windows (the ROM's Fire2/RET keys — Space is not a text key; a press mid-print SKIPS
  to the next page, like SkipText) · B file a bug report (dev/QA, see below).
- BUG REPORT (B) — a DEV/QA tool, NOT a ROM feature. A rolling pair of MediaRecorders keeps the
  last ~20s of gameplay (game.js initBugReporter: two recorders ping-pong 40s windows staggered by
  20s, so one always holds >=20s ending now as a single playable .webm). The recorded stream is the
  canvas video PLUS the audio-bus tap, so clips HAVE SOUND: all sound now connects via audioOut() to
  a master GainNode (audioBus) that fans out to the speakers and a MediaStreamAudioDestination
  (captureDest), both created with the AudioContext in unlockAudio; bugRefreshAudio() restarts the
  recorders once audio unlocks so segments include the track. Pressing B freezes the clip, then a
  small overlay form (#report-form) lets the user add a short description or cancel (Esc cancels,
  Ctrl/⌘+Enter submits; the form stops keystrokes reaching the game handlers). On submit the clip +
  metadata + description POST to serve.js `/report`, which uploads the clip as an asset on the
  `bug-clips` release and opens a GitHub issue (title/body include the description), via the GitHub
  REST API. Setup: copy web/.env.example to web/.env (gitignored) and add a fine-grained PAT
  (Contents + Issues: read/write on the repo); override the target with GITHUB_REPO (default
  southernsun/MetalGearJS). An on-canvas toast reports the issue number/link or the error; no token →
  serve.js returns a helpful 500. serve.js honors PORT and returns 204 for /favicon.ico.
- PASSWORDS + SAVE/LOAD (passwords.asm; ChkPasswords on ExitPauseMode): while PAUSED, TYPE a code,
  then unpause to apply it. The ROM's five cheats: DS 4 (class +2) · ANTA WA ERAI (class +1) ·
  INTRUDER (max ammo) · ISOLATION (max rations) · HIRAKE GOMA (all 8 cards + Grey Fox's cell). The
  same pause-typed input also drives SAVE / LOAD: the ROM persists GameProgressBuffer to CASSETTE
  TAPE (no browser equivalent — moved to coverage outOfScope), so a faithful localStorage analog
  serialises the same progress (serializeProgress/restoreProgress) — type SAVE or LOAD in pause.
- Demo cluster map (room 0 = start, bottom-left; many shared edges are WALLED — the building
  is a maze; `node Tools/check-graph.mjs` computes GLOBAL footprint reachability from spawn
  and prints how every room is reached + walkable spots; the headless suites validate every
  demo placement against it):
  ```
  row 3:   3   7  11        3-7 connects; 7's doors gate 6's main floor and 11
  row 2:   2   6  10  14    rooms 12-14 are NOT reachable on foot in this cluster
  row 1:   1   5   9  13    6's main floor, 10 and 9 are ONLY behind the card doors
  row 0:   0   4   8  12    121 lies below 0; water/ladder rooms are ?room= dev hooks
  ```
  ALL card doors here (6<->7 and 7<->11) are lock 5 = **CARD4** (ChkCard maps lock L to card
  L-1; CARD5 fits the water rooms' lock-6 doors instead). Select CARD4 (E menu or I), face
  the door, and push into it. Entering room 6 from 5 only reaches its bottom corridor; the
  main floor (and onward to 10 -> 9) is entered through room 7's south door. Room 8's
  walkable area is only its bottom-left quarter (entered from 4). Room 3's NORTH door is the
  ELEVATOR (push up into it): room 3 ⇄ elevator room 240 ⇄ room 31 (the top floor, exported
  as an island) — stand in the cabin and hold up/down; it stops at each floor; walk right
  through the floor opening to exit.

## Key decode notes (hard-won)
- Guard sprite (`SprGuard`, 32 sprites): sprites 0–7 = upper bodies OR-paired per direction
  `(0,1)=down (2,3)=left (4,5)=up (6,7)=right`; sprites 8–31 = legs, OR-pairs at `8+phase*8+2*dir`
  (3 leg phases). Frame = upper[dir] over lower[dir][phase]. Colours = `ActorSprColors3` (slot 2 =
  uniform, CC|0Dh = face). guard.png is baked ONE palette, then RECOLOURED per room at draw time
  (SetSprPal: SpritesetRooms→SprsetPal slots 2/0x0D) — see "Guard fidelity pass (2026-06-15)" below.
- Guard frame identity (verified in actorspriteattr.asm + guard.asm): phase 0 = GuardWalk1,
  phase 1 = the standing `Guard<Dir>` frame (GuardPatrolTurn sets SpriteId = Direction + 8),
  phase 2 = GuardWalk2. Walk cycle toggles phase0↔phase2; the standing frame shows only while
